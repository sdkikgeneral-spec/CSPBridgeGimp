# CSP × GIMP プラグインブリッジ
## 技術調査・ライセンス確認まとめ

---

## 1. 概要

Clip Studio Paint EX のプラグインから GIMP プラグインをブリッジ実行できないかを検討した。Photoshop プラグインは Adobe 独自の閉じたアーキテクチャのため非現実的だが、GIMP はオープンソースであり技術的に実現可能と判断した。

---

## 2. Photoshop プラグインとの比較

| 項目 | Photoshop プラグイン | GIMP プラグイン |
|---|---|---|
| アーキテクチャ | Adobe 独自 SDK（非公開） | オープンソース |
| プラグイン形式 | DLL（.8bf） | スタンドアロン EXE/ELF |
| ホスト通信 | Adobe 固有コールバック | Wire Protocol（IPC） |
| ソース参照 | 不可 | 完全に参照可能 |
| ブリッジ実現性 | ✕ 非現実的 | ◎ 実現可能 |

---

## 3. ブリッジアーキテクチャ

### 3.1 基本構成

GIMP プラグインはスタンドアロンの実行ファイルであり、GIMP 本体と IPC（プロセス間通信）で通信する Wire Protocol を使用する。CSP プラグイン側でこのホスト役をエミュレートすることで、GIMP プラグインをそのまま子プロセスとして動作させられる。

### 3.2 処理フロー

```
CSPプラグイン（C++）
    ↓ 選択範囲・画像バッファを取得
    ↓ 一時バッファ or 共有メモリに書き出し
    ↓ GIMPプラグイン実行ファイルを子プロセス起動
    ↓ Wire Protocol でパイプ通信（画像タイル送受信）
    ↓ 処理結果バッファを CSP レイヤーに書き戻し
```

- CSP プラグイン（C++）が選択範囲・画像バッファを取得
- 画像データを一時バッファまたは共有メモリに書き出し
- GIMP プラグイン実行ファイルを子プロセスとして起動
- Wire Protocol でパイプ通信（画像タイル送受信）
- 処理結果バッファを CSP のレイヤーに書き戻し

## 4. 使用ライブラリ構成

| ライブラリ | 役割 | そのまま使用 | 備考 |
|---|---|---|---|
| libgimpwire | Wire Protocol のシリアライズ/デシリアライズ | ✅ 可能 | ホスト実装の中核 |
| libgimpbase | 基本型・パラメータ定義 | ✅ 可能 | 型定義のみ |
| libgimp | プロシージャ DB・コールバック | ⚠️ 部分的 | ホスト側スタブが必要 |
| タイル転送層 | 画像データのやり取り | ❌ 自前実装 | gimp_tile_get/put |

### 4.1 最小スタブ実装が必要なプロシージャ（フィルター系のみ）

- `gimp_image_list()`
- `gimp_drawable_get()`
- `gimp_tile_get()` / `gimp_tile_put()` ← 核心部分
- `gimp_display_list()`
- UI 系コール（ダイアログ等）→ スタブで無視

## 5. ライセンス確認結果

### 5.1 各ライブラリのライセンス（公式リポジトリ確認済み）

| ライブラリ | ライセンス | 商用プラグインへの影響 |
|---|---|---|
| libgimpwire | LGPL v3 | 動的リンクなら非公開・商用販売 OK |
| libgimpbase | LGPL v3 | 同上 |
| libgimp | LGPL v3 | 同上 |
| GIMP コア・app | GPL v3 | 直接リンク不要のため無関係 |

### 5.2 LGPL v3 の条件

- 動的リンク（DLL）で使用すること
- CSP プラグイン本体のソースは非公開・商用販売 OK
- libgimp 側を改変した場合はその差分のみ公開義務あり
- 無改変ならば義務なし

### 5.3 推奨ライブラリ構成

| モジュール | 内容 | ライセンス対応 |
|---|---|---|
| libgimpwire.dll | 無改変でそのまま同梱 | LGPL 準拠（差分なし） |
| libgimpbase.dll | 無改変でそのまま同梱 | LGPL 準拠（差分なし） |
| MyGimpHost.dll | 自前スタブ（完全オリジナル） | 非公開・商用 OK |

## 6. 実装コスト見積もり

| 作業 | コスト | 補足 |
|---|---|---|
| Wire Protocol 解析（libgimpwire） | 中 | ソースが公開されているため参照可能 |
| ホストエミュレーター C++ 実装 | 高 | フィルター系のみに絞れば低減可能 |
| 画像タイル形式の変換 | 低 | RGBA バッファ変換のみ |
| UI 系コールのスタブ化 | 中 | ダイアログ等を無視する実装 |

---

## 7. 結論と推奨アクション

技術・ライセンスの両面から実現可能と判断する。一度ブリッジを実装すれば大量の GIMP プラグインが動く汎用基盤となり、投資対効果は高い。

| 観点 | 評価 | コメント |
|---|---|---|
| 技術的実現性 | ◎ | Wire Protocol がオープンで参照実装可能 |
| ライセンス | ◎ | LGPL のため商用プラグインとして販売可能 |
| 実装コスト | △ | ホストエミュレーター部分が重いが一度限り |
| 汎用性 | ◎ | 完成後は多数の GIMP プラグインが利用可能 |

> 推奨する進め方：libgimpwire + libgimpbase をそのままビルドに取り込み、libgimp のホスト側のみ最小スタブで自前実装する。まずフィルター系プラグイン 1 本を動かすプロトタイプから着手することを推奨する。

## 8. 公式ドキュメント

ソース解析なしでもブリッジ設計が可能なドキュメントが `developer.gimp.org` に公式整備されている。

### 8.1 利用可能なドキュメント一覧

| ドキュメント | 内容 | URL |
|---|---|---|
| GIMP 3.0 API リファレンス | libgimp 全 API の公式リファレンス | developer.gimp.org/resource/api/ |
| How to write a plug-in | プラグインアーキテクチャのチュートリアル（C / Python / Script-Fu） | developer.gimp.org/resource/writing-a-plug-in/ |
| About Plug-ins and Filters | プラグイン・フィルターの種別・契約・run mode の解説 | developer.gimp.org/resource/about-plugins/ |
| Debugging Plug-ins | プラグインデバッグガイド | developer.gimp.org |
| Distributing plug-ins | 配布・パッケージングガイド | developer.gimp.org |

### 8.2 ブリッジ実装に特に重要な情報

- Wire Protocol は「the wire」として公式に文書化されており、IPC 通信の仕様が参照可能
- run mode（`GIMP_RUN_NONINTERACTIVE`）を使えば、ブリッジ時に GUI ダイアログのスタブ実装が不要になる可能性あり
- libgimp API リファレンスから `gimp_tile_get` / `gimp_tile_put` 等の仕様を直接確認可能
- 公式チュートリアルに C 言語サンプルコードあり（プロトタイプ実装の参考になる）

### 8.3 実装コスト再評価

公式ドキュメントの存在により、当初「中〜高」と見積もった Wire Protocol 解析コストは大幅に低減できる。

| 作業 | 当初コスト | ドキュメント活用後 |
|---|---|---|
| Wire Protocol 解析 | 中 | 低（公式文書化済み） |
| ホストエミュレーター実装 | 高 | 中（API リファレンスで仕様明確） |
| run mode スタブ化 | 中 | 低〜不要（非インタラクティブモード利用） |
| 画像タイル変換 | 低 | 低（変わらず） |
